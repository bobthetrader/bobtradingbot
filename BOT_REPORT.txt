1. OVERVIEW
================================================================================

A fully automated, multi-pair, multi-exchange cryptocurrency trading bot that:
- Trades BTC, ETH, SOL, XRP, ADA, DOT, LINK against EUR on Kraken
- Mirrors strong signals on Alpaca (MSTR, COIN, MARA correlated stocks)
- Detects and trades new Kraken listings within 60 minutes of listing
- Uses 5 AI models + 4 data streams to assess market conditions
- Self-tunes parameters using the scientific method (one change at a time)
- Pre-screens parameter changes with a 30-day per-pair backtest before deploying
- Sends Telegram notifications for every trade
- Displays all data on a live web dashboard (Railway public URL)
- Has a local Docker standby for Railway failover

3. TRADING STRATEGY & SIGNAL GENERATION
================================================================================

Signal Engine (analysis.py)
-----------------------------
Generates a (signal, score) tuple for each pair every loop:

MEAN REVERSION PATH (per-pair RSI thresholds, see Section 4):
  RSI oversold  → BUY signal,  score proportional to distance from threshold
  RSI overbought → SELL signal, score proportional to distance from threshold

TREND/BREAKOUT PATH:
  Price above Bollinger upper band + RSI confirmation → BUY
  Price below Bollinger lower band + RSI confirmation → SELL

SMA MOMENTUM PATH (always active, fills gaps when RSI/BB are neutral):
  SMA8 > SMA21 → BUY bias with score proportional to gap (multiplier ×50)
  SMA8 < SMA21 → SELL bias
  Ensures the bot always has a signal even in quiet markets

SIGNAL SELECTION:
  Each pair is evaluated independently. The pair with the highest absolute
  score among actionable signals is selected each loop.
  Contradictory signals (BUY signal + negative score) are filtered out.
  Non-actionable SELLs (no position + shorts disabled) are skipped so they
  don't beat a genuine BUY signal.

Price history: 200 candles per pair, seeded from 15-minute Kraken OHLC on
startup so RSI/SMA work immediately without a cold-start delay.

4. PER-PAIR STRATEGY PROFILES
================================================================================

Each coin gets its own RSI thresholds and minimum signal score:

  XBTEUR / ETHEUR   Strategy: Trend-following
                    RSI buy: 32 | RSI sell: 68 | Min score: 8
                    Rationale: Large caps move in clearer trends,
                               need stronger confirmation before entry

  SOLEUR / XRPEUR / ADAEUR   Strategy: Mean reversion
                               RSI buy: 28 | RSI sell: 72 | Min score: 5
                               Rationale: Mid caps overshoot RSI extremes,
                                          reversals are more reliable

  DOTEUR / LINKEUR   Strategy: Mean reversion
                     RSI buy: 25 | RSI sell: 75 | Min score: 3
                     Rationale: Smaller caps have bigger swings,
                                even RSI 25 is worth acting on


================================================================================
5. RISK MANAGEMENT & POSITION SIZING
================================================================================

Seven multipliers are applied in sequence to the base allocation:

  BASE          allocation_per_trade_percent × available_eur
  × VOLATILITY  ATR scaling (target volatility per trade)
  × REGIME      risk_off_allocation_multiplier when regime is RISK_OFF
  × BREAKOUT    2× for fresh Bollinger breakout (<25 days), 0.5× if stale
  × CORRELATION 1/(1+n_correlated_open) — prevents stacking correlated bets
                All crypto pairs are treated as one correlation group
                0 open: 1.00× | 1 open: 0.67× | 2 open: 0.50× | 3: 0.33×
  × REGIME_SIZE 1.2× trending markets (bigger bets), 0.7× ranging markets
  × KELLY       Half-Kelly fraction from actual win rate and avg win/loss
                Updates after every closed trade. Needs 10+ trades.
                Maps kelly [0.01-0.5] to multiplier [0.3-1.5]
  × MONTHLY     0.3× when monthly return >8% (protect gains)
                0.7× when 3-8% (on track, slightly conservative)
                1.0× when 0-3% (normal)
                1.2× when negative (slight aggression)

Stop Loss Mechanisms (3 layers):
  1. SOFT STOP   stop_loss_percent — regular exit on position loss
  2. HARD STOP   hard_stop_loss_percent — force-close backstop
  3. BREAK-EVEN  After +0.5% move, stop moves to entry price

Intelligence-Adjusted Buy Gate:
  effective_min = pair_min_score - (intelligence_score × score_weight)
  Bullish AI → lower gate (easier to buy)
  Bearish AI → higher gate (harder to buy)

6. MARKET REGIME DETECTION
================================================================================

Three-state classification using SMA8/21/50 on BTC price history:

  TRENDING_UP    SMA8 > SMA21 > SMA50, gap ≥ 0.3%
                 Strategy: trend-following only, mean reversion disabled
                 Position size: 1.2×

  TRENDING_DOWN  SMA8 < SMA21 < SMA50, gap ≥ 0.3%
                 Strategy: trend-following only
                 Position size: 0.8×

  RANGING        SMAs tangled, no clear alignment
                 Strategy: mean reversion only, trend breakout disabled
                 Position size: 0.7×

BTC downtrend flag (separate from above):
  True when BTC close < 20-period SMA
  Used for: two-tier short sizing (HEDGE 3% NAV vs BEAR 5% NAV)

9. SELF-LEARNING SYSTEMS
================================================================================

A. SCIENTIFIC METHOD PARAMETER OPTIMIZER (core/param_optimizer.py)
──────────────────────────────────────────────────────────────────
Process:
  1. Establish Sharpe ratio baseline from closed trade history
  2. Pre-screen with 30-day backtest (see below)
  3. Change exactly ONE parameter by ONE step (up or down)
  4. Wait for MIN_EVAL_TRADES (10) new closed trades
  5. Measure new Sharpe — KEEP if improved, REVERT if not
  6. Advance to next parameter

Parameters cycled:
  min_buy_score (5-50, step 2), take_profit_percent (1-6%, step 0.25)
  stop_loss_percent (1.5-10%, step 0.5), mr_rsi_oversold (20-45, step 2)
  mr_rsi_overbought (50-75, step 2), volume_filter_min_ratio (0.2-2, step 0.1)
  allocation_per_trade_percent (5-70%, step 5)

B. BACKTEST GATE (before any parameter change is deployed)
──────────────────────────────────────────────────────────
Each pair is backtested independently using its own RSI profile:
  - BTC/ETH: tested with their trend-following RSI thresholds (32/68)
  - SOL/XRP/ADA: tested with mean-reversion thresholds (28/72)
  - DOT/LINK: tested with small-cap thresholds (25/75)
  - RSI changes only test pairs using mean reversion strategy
  - TP/SL changes test all pairs

Data: Kraken OHLC API (240-minute candles × 180 = 30 days, cached 1 hour)
Threshold: rejects change if avg backtest Sharpe < baseline - 0.2
Result: shows which pairs helped vs hurt by the change

C. KELLY CRITERION POSITION SIZING
────────────────────────────────────
Kelly = win_rate - ((1 - win_rate) / avg_win_loss_ratio)
Half-Kelly applied (0.5× Kelly) for lower variance
Updates after every closed trade
Needs 10+ closed trades to activate (defaults to 0.1 before then)

D. AI CALIBRATION FROM HISTORY
────────────────────────────────
All AI models receive their own past decisions and outcomes:
  "Last time score was -3, BTC dropped 2% — correct call"
  "Score was +2, ETH fell 1% — incorrect call"
Models calibrate over time based on what actually happened.

E. SHARPE RATIO TRACKING
────────────────────────
Goal thresholds: Success ≥ 3.0 | Failure < 1.0 | Neutral 1.0-3.0
Calculated from data/trade_events_paper.jsonl or trade_events_live.jsonl
Minimum 5 closed trades for calculation
Updated every 5 closed trades
Initialised from file on startup (survives restarts)

Separate journals for paper and live mode — live Sharpe is never
diluted by paper trading data.

10. NEW LISTINGS MONITOR (core/listings_monitor.py)
================================================================================

Three detection sources (polled every 10 minutes):

  1. Kraken AssetPairs API (fastest — detects the moment a pair goes live)
     GET https://api.kraken.com/0/public/AssetPairs
     Compares against saved baseline, alerts on new EUR pairs
     Free, no key required

  2. Kraken Blog RSS (early warning — often hours before trading starts)
     GET https://blog.kraken.com/feed
     Parses listing keywords, extracts coin symbols
     Free, no key required

  3. Sharpe.ai /v1/listings/recent (hourly aggregated)
     Requires SHARPE_API_KEY

Strategy for each detected listing:
  T+0:     Detection → Telegram alert → record price
  T+60min: Start checking if price is trending up
  +2%:     BUY (up to trade_amount_eur size)
  +12h:    FORCE SELL regardless of P&L (exit the pump before the dump)

Pair name auto-resolution:
  Tries COINEUR, XCOINZEUR, XCOINEUR in order until Kraken returns price
  If no price found → skip safely

State persisted in data/listing_watchlist.json (survives restarts).
Dashboard card shows: status, price vs detection, qty held, P&L, 12h timer.

11. ALPACA CORRELATED STOCKS (core/alpaca_interface.py)
================================================================================

Trades crypto-correlated stocks on Alpaca when strong Kraken signals fire.

Strategy:
  BTC BUY signal (score ≥ 10) → buy MSTR, COIN, MARA on Alpaca
  ETH BUY signal (score ≥ 10) → buy COIN on Alpaca
  SELL signal on same pair    → close the Alpaca positions

Rationale:
  BTC price moves tend to flow into crypto-proxy stocks 1-3 hours later.
  This captures the lag between crypto and stock market reaction.
  Alpaca is commission-free; positions are fractional shares.

Sizing: 5% of Alpaca portfolio value per correlated stock
Only trades when US market is open (checked via /v2/clock)
Sends separate Telegram notifications for Alpaca trades.

Configuration:
  ALPACA_API_KEY      — from alpaca.markets
  ALPACA_API_SECRET   — same
  ALPACA_BASE_URL     — https://paper-api.alpaca.markets (paper)
                        https://api.alpaca.markets (live)

================================================================================

SQLite database at data/bot_history.db — persisted on Railway volume.
Survives all restarts and redeploys. Max 5,000 rows per table.

Tables:
  trades              Every trade (type, pair, volume, price, P&L, balance)
  ai_panels           Every AI panel result with per-model scores + outcome
  sharpe_snapshots    Funding rates, insider signal, OI per refresh
  bot_snapshots       Balance, regime, signals, Sharpe every 10 loops

they can see whether their past calls were correct. This is the core
of the AI learning loop.

15. LOCAL DOCKER STANDBY
================================================================================

Provides hot standby in case Railway goes down.

Setup:
  1. Install Docker Desktop (docker.com)
  2. Enable "Start Docker Desktop when you log in" in Docker settings
  3. cd D:\tradingbot && docker-compose up -d
  4. Dashboard available at http://localhost:8080

docker-compose.yml:
  - Always starts in --paper mode (BOT_ARGS: "--paper")
  - Persistent local volume for data
  - restart: unless-stopped (auto-restarts on PC reboot)

Failover procedure:
  1. Run: python monitor_railway.py
     Checks Railway every 5 minutes, Telegram alert if 2+ consecutive failures
  2. If alert received: docker-compose up -d
  3. Local bot is in paper mode — to switch live:
     Edit docker-compose.yml: set BOT_ARGS: "" and docker-compose restart
  4. When Railway recovers: docker-compose down

CRITICAL: Never run both Railway AND local in live mode simultaneously.
          They would both place orders on the same Kraken account.
