1. OVERVIEW
================================================================================

A fully automated, multi-pair, multi-strategy cryptocurrency trading bot that:
- Trades BTC, ETH, SOL, XRP, ADA, DOT, LINK against EUR on Kraken (main bot)
- Runs a concurrent scalping engine across 10 pairs (BTC, ETH, SOL, XRP, LINK,
  AVAX, ADA, DOT, ATOM, UNI) with dynamic fee-aware take-profit targets
- Uses Ichimoku Cloud + Gaussian Channel to filter buy signals by trend
- Uses 5 AI models + 4 data streams to assess market conditions
- Wires LunarCrush sentiment and on-chain data directly into the buy gate
- Fires a circuit breaker when drawdown exceeds limit — closes all positions
- Self-tunes parameters using the scientific method (one change at a time)
- Pre-screens parameter changes with a 30-day per-pair backtest
- Sends Telegram notifications for every trade
- Displays all data on a live web dashboard (bobtradingbot.com)
- Deployed 24/7 on Hetzner VPS with Caddy reverse proxy + Cloudflare SSL

3. TRADING STRATEGY & SIGNAL GENERATION
================================================================================

Signal Engine (analysis.py)
-----------------------------
Generates a (signal, score) tuple for each pair every loop:

MEAN REVERSION PATH (per-pair RSI thresholds, see Section 4):
  RSI oversold  → BUY signal, score proportional to distance from threshold
  RSI overbought → SELL signal, score proportional to distance from threshold

TREND/BREAKOUT PATH:
  Price above Bollinger upper band + RSI confirmation → BUY
  Price below Bollinger lower band + RSI confirmation → SELL

SMA MOMENTUM PATH (always active):
  SMA8 > SMA21 → BUY bias, score proportional to gap (multiplier ×50)
  SMA8 < SMA21 → SELL bias
  Ensures the bot always has a signal even in quiet markets

SIGNAL SELECTION:
  Each pair evaluated independently. Highest absolute score among actionable
  signals selected each loop. Contradictory signals filtered out.
  Non-actionable SELLs (no position + shorts disabled) skipped.

Price history: 200 candles per pair, seeded from 60-minute Kraken OHLC on
startup so RSI/SMA work immediately without a cold-start delay.

5. RISK MANAGEMENT & POSITION SIZING
================================================================================

Seven multipliers applied in sequence to the base allocation:

  BASE          allocation_per_trade_percent × available_eur
  × VOLATILITY  ATR scaling (target volatility per trade)
  × REGIME      risk_off_allocation_multiplier when regime is RISK_OFF
  × BREAKOUT    2× for fresh Bollinger breakout (<25 days), 0.5× if stale
  × CORRELATION 1/(1+n_correlated_open) — prevents stacking correlated bets
                0 open: 1.00× | 1 open: 0.67× | 2 open: 0.50× | 3: 0.33×
  × REGIME_SIZE 1.2× trending markets, 0.7× ranging markets
  × KELLY       Half-Kelly fraction from actual win rate and avg win/loss ratio
                Updates after every closed trade. Needs 10+ trades.
  × MONTHLY     0.3× when monthly return >8% | 0.7× when 3-8% | 1.0× normal
                1.2× when negative (slight aggression to recover)

Stop Loss Mechanisms (3 layers):
  1. SOFT STOP   stop_loss_percent — regular exit on position loss
  2. HARD STOP   hard_stop_loss_percent — force-close backstop
  3. BREAK-EVEN  After +0.5% move, stop moves to entry price

Buy Gate Adjustments (stacked):
  effective_min = pair_min_score
                - (intelligence_score × score_weight)     [AI panel]
                - (lunarcrush_combined × 0.5)             [social sentiment]
                - (onchain_combined × 0.5)                [on-chain data]
                + ichimoku_gaussian_score_boost           [if near Gaussian band]

  Bullish signals lower the gate (easier to buy)
  Bearish signals raise the gate (harder to buy)
  Ichimoku inside/below cloud → buy blocked entirely

6. SHORT POSITION MANAGEMENT
================================================================================

File: trading_bot.py (execute_open_short_order / execute_close_short_order)
Config: config.paper.toml [shorting]
Requires: Kraken margin trading enabled on account

OVERVIEW
  The bot opens leveraged short positions (2× leverage, max €30 notional) when
  a pair has a strong bearish signal and the trend is confirmed bearish. Shorts
  are tracked separately from longs in short_qty / short_entry_prices dicts and
  persisted in purchase_prices_paper.json with side="short".

KRAKEN MARGIN FEES (applied to all shorts)
  Opening trade fee (taker):  0.26%
  Opening margin position fee: 0.02%
  Rollover fee:               0.02% every 4 hours
  Closing trade fee (taker):  0.26%
  Total fees for 12h hold:    ~0.62% (3 rollovers)

EXIT CONDITIONS (priority order)
  1. SHORT_TAKE_PROFIT   Price falls ≥ 1.1% from entry (covers ~0.62% fees + 0.5% buffer)
  2. SHORT_STOP_LOSS     Price rises ≥ 0.5% from entry — cut loss immediately
  3. SHORT_TIME_REVIEW   After 12h, every loop: if net P&L (gross − position fees) is
                         below the cost of the next 4h rollover (0.02%), close the short.
                         Formula: net_pnl = gross_change% − (0.04% + n_rollovers × 0.02%)
                         Close if net_pnl < 0.02%

TIME REVIEW LOGIC
  At T+12h the bot starts checking whether holding another 4h still makes sense.
  Position fees accrue as: open_fee(0.02%) + n_rollovers × 0.02% + close_fee(0.02%)
  If current net P&L (after position fees) < one more rollover (0.02%), close now.
  This fires every loop after 12h — naturally re-evaluates at each 4h rollover tick.

MANUAL CLOSE
  Dashboard endpoint: /close-short/<pair>
  Creates FORCE_SHORT_CLOSE_<pair> flag file in data volume.
  Bot detects on next loop and calls execute_close_short_order at current price.

CONFIGURATION (config.paper.toml)
  [shorting]
  enabled                  = true
  leverage                 = 2
  max_short_notional_eur   = 30.0
  short_take_profit_percent = 1.1   (covers 12h fees + 0.5% buffer)
  short_stop_loss_percent  = 0.5    (cut if price moves 0.5% against short)

STATE PERSISTENCE
  Open shorts stored in purchase_prices_paper.json with side="short", qty, entry_price_eur,
  entry_ts. Loaded back on restart so short_qty / short_entry_prices are correctly populated.
  entry_timestamps[pair] used for time review (Unix seconds since short opened).

7. MARKET REGIME DETECTION
================================================================================

Three-state classification using SMA8/21/50 on BTC price history:

  TRENDING_UP    SMA8 > SMA21 > SMA50, gap ≥ 0.3%
                 Strategy: trend-following only, mean reversion disabled
                 Position size: 1.2×

  TRENDING_DOWN  SMA8 < SMA21 < SMA50, gap ≥ 0.3%
                 Strategy: trend-following only
                 Position size: 0.8×

  RANGING        SMAs tangled, no clear alignment
                 Strategy: mean reversion only, trend breakout disabled
                 Position size: 0.7×

BTC downtrend flag (separate from above):
  True when BTC close < 20-period SMA
  Used for: two-tier short sizing (HEDGE 3% NAV vs BEAR 5% NAV)

8. ICHIMOKU CLOUD + GAUSSIAN CHANNEL FILTER
================================================================================

File: core/ichimoku_gaussian.py
Data: 1-hour OHLC candles from Kraken REST API
Cache: 5-minute TTL per pair (no extra REST calls per loop)

ICHIMOKU CLOUD (non-standard 20/30/60 cycle — faster than traditional 9/26/52)
  Tenkan-sen:   (20-period high + low) / 2   (conversion line)
  Kijun-sen:    (30-period high + low) / 2   (base line)
  Senkou Span A: (Tenkan + Kijun) / 2         (leading span A)
  Senkou Span B: (60-period high + low) / 2   (leading span B)
  Cloud = area between Span A and Span B

Price position determines trade gate:
  ABOVE cloud   → uptrend confirmed, buys allowed
  INSIDE cloud  → choppy/indecisive, BUY BLOCKED
  BELOW cloud   → downtrend, BUY BLOCKED

GAUSSIAN CHANNEL (30-period, 2.0× standard deviation)
  Gaussian-weighted moving average (smooth, noise-reduced)
  Upper band = GMA + 2.0 × std
  Lower band = GMA - 2.0 × std

  Buy signal: price is within 0.5% of the lower band (touching, not crashed)
  Only signals when price is AT the band, not when it has broken through

COMBINED RULES
  1. Ichimoku inside or below cloud → block buy regardless of other signals
  2. Ichimoku above cloud + Gaussian lower band touch → score boost +1.5
  3. Ichimoku above cloud, no Gaussian signal → existing signals proceed normally

AI INTEGRATION
  All 5 AI models receive per-pair Ichimoku + Gaussian data in their prompts
  so they can reason about trend structure and calibrate scores accordingly.

DASHBOARD CARD
  Shows per-pair: cloud position (ABOVE/INSIDE/BELOW), trend, cloud price range,
  Gaussian buy signal, score boost

10. SCALPER AI PARAMETER TUNER
================================================================================

File:    core/scalper_ai.py
Trigger: Every 25 closed scalper trades
Models:  Llama 3.1 8B → DeepSeek Chat → Hermes 3 70B (cascade, cheapest first)

OVERVIEW
  A/B experiment system that tunes scalper parameters one at a time using recent
  trade data. Each cycle: evaluate the previous experiment, then propose the next
  single-parameter change to test for the following 25 trades.

PARAMETERS TUNED
  rsi_buy      (25–40)   RSI threshold to enter a long scalp
  rsi_sell     (60–75)   RSI threshold to exit
  vwap_thresh  (0.001–0.006)  % deviation from VWAP required to signal
  score_thresh (1.5–3.0) Minimum combined signal score to trade
  sl_pct       (0.15–0.35%)   Stop-loss percentage

ANALYSIS WINDOW
  75 trades (3 full cycles) — gives a broader market picture before proposing
  changes. Experiment evaluation always uses exactly 25 trades (clean A/B).

BLACKLIST (pairs to exclude from scalping)
  - Rebuilt fresh each AI cycle — never inherited from previous state
  - AI suggests pairs with ≤33% WR in the current 75-trade window
  - Hard guard: pairs with >50% WR in the window CANNOT be blacklisted
    regardless of what the AI suggests
  - Cap: maximum 5 pairs blacklisted at any time
  - Cap logic: if AI suggests >5, use actual worst performers from window data
  - If overall WR < 40%, AI is instructed to blacklist nothing (market problem, not pairs)

EXPERIMENT EVALUATION
  After 25 trades: compare experiment WR vs baseline WR
  Within 5 percentage points of baseline → KEPT (neutral tolerance)
  More than 5pp worse → REVERTED, direction marked as failed for 24h

FAILED CHANGE MEMORY
  A parameter direction that was tried and reverted is blocked for 24 hours.
  After 24h it is eligible again (crypto regime can shift hourly).
  Max 10 failed changes remembered at any time (rolling window).

3-CYCLE TRAJECTORY CONTEXT IN PROMPT
  The last 3 experiment cycles are included in the AI prompt showing: type
  (propose/evaluate), WR achieved, what changed, and the AI's reasoning.
  This lets the AI see the DIRECTION things are moving, not just the current state.

BACKTEST INSIGHTS INJECTED INTO PROMPT
  On each AI trigger, top 3 statistically validated signal combinations from
  backtest_recommendations.json are injected into the prompt as BACKTEST INSIGHTS.
  Includes Wilson CI, Bayesian probability, Kelly fraction, BH-significance flag.
  AI uses these to guide which direction to nudge parameters.

DASHBOARD DISPLAY
  Adjustment History card shows: timestamp, WR, parameter changes, AI reasoning.
  Current blacklisted pairs shown at top of scalper card.

13. SELF-LEARNING SYSTEMS
================================================================================

A. SCIENTIFIC METHOD PARAMETER OPTIMIZER (core/param_optimizer.py)
──────────────────────────────────────────────────────────────────
Process:
  1. Establish Sharpe ratio baseline from closed trade history
  2. Pre-screen with 30-day backtest (Kraken OHLC, conservative slippage)
  3. Change exactly ONE parameter by ONE step (up or down)
  4. Wait for eval_every_n_trades (10) new closed trades
  5. Measure new Sharpe — KEEP if improved, REVERT if not
  6. Advance to next parameter

Parameters cycled:
  min_buy_score, take_profit_percent, stop_loss_percent,
  mr_rsi_oversold, mr_rsi_overbought, volume_filter_min_ratio,
  allocation_per_trade_percent

Optimizer activates after first 10 closed trades.

B. BACKTEST GATE (before any parameter change is deployed)
──────────────────────────────────────────────────────────
Each pair backtested independently using its own RSI profile.
Data: Kraken OHLC API (240-minute candles × 180 = 30 days, cached 1 hour)
Exit slippage correctly applied to conservative backtest exits.
Threshold: rejects change if avg backtest Sharpe < baseline - 0.2

C. KELLY CRITERION POSITION SIZING
────────────────────────────────────
Kelly = win_rate - ((1 - win_rate) / avg_win_loss_ratio)
Half-Kelly applied (0.5×) for lower variance
Updates after every closed trade. Needs 10+ trades to activate.

D. AI CALIBRATION FROM HISTORY
────────────────────────────────
All AI models receive their own past decisions and outcomes in every prompt.
Models calibrate over time based on what actually happened.
Ichimoku + Gaussian data included so models learn trend context vs outcomes.

E. SHARPE RATIO TRACKING
────────────────────────
Goal thresholds: Success ≥ 3.0 | Failure < 1.0 | Neutral 1.0-3.0
Minimum 5 closed trades for calculation. Updated every 5 closed trades.
Separate journals for paper and live mode.
Scalper trades tracked separately — not mixed into main bot Sharpe.

F. SCALPER AI PARAMETER TUNER  →  see Section 10 for full details
────────────────────────────────
Every 25 scalper trades: AI evaluates last experiment, proposes next single-param
change using a 75-trade (3-cycle) window. Blacklist rebuilt fresh each cycle.

G. STATISTICAL BACKTEST → SCALPER AI FEEDBACK LOOP
────────────────────────────────────────────────────
File:    backtest/scalper_backtest.py  (standalone, run locally)
Method:  Wilson Score CI + Bayesian Beta-Binomial + Kelly Criterion + BH FDR correction
         Tests all combinations of rsi_buy_max and score_min from real scalper_trades.jsonl.
         Filters to BH-significant combinations, ranks by reliability.

Output:  data/backtest_recommendations.json (auto-committed to git after each run)
         Contains top 3 validated signal combos with Wilson CI, P(WR>50%), Kelly fraction.

Daily pipeline (runs at 09:30–09:45 UTC):
  09:30  Server cron extracts scalper_trades.jsonl from Docker volume
  09:35  Windows Task Scheduler runs scripts/daily_backtest.ps1 on local PC:
           - SSHes to server with restricted key (id_ed25519_botauto)
           - Pulls scalper_trades.jsonl to local
           - Runs statistical backtest locally
           - SSHes back to push backtest_recommendations.json to server
  09:45  Server cron runs server_daily_pull.sh:
           - git pull (picks up recommendations)
           - Copies recommendations to Docker data volume
  Bot:   Reads recommendations on next AI trigger — no restart needed.

SSH security: restricted key (id_ed25519_botauto, no passphrase) limited via
  command= in /root/.ssh/authorized_keys to only run extract or pull commands.
  Admin key (id_ed25519, passphrase) used for manual SSH sessions.

16. CIRCUIT BREAKER
================================================================================

Protects capital during severe drawdown events.

Trigger:
  When peak-to-current portfolio drawdown >= max_drawdown_percent (default 10%)

Actions on trigger:
  1. Force-close ALL open positions immediately (market sell)
  2. Send Telegram alert with drawdown % and portfolio values
  3. Pause all buying for 24 hours
  4. Show red warning banner on dashboard

Reset:
  Auto-resets when drawdown recovers below 50% of the threshold
  (e.g. if limit is 10%, resets when drawdown recovers to <5%)

Config: risk_management.max_drawdown_percent in config.paper.toml
Dashboard: Red banner appears at top of page when circuit breaker is active
