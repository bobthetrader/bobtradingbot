1. OVERVIEW
================================================================================

This is a fully automated, multi-pair cryptocurrency trading bot that trades
BTC, ETH, SOL and XRP against EUR on the Kraken exchange. It operates 24/7
on Railway cloud infrastructure and features:

- Real-time market data via Kraken WebSocket feed
- Technical analysis signal engine (RSI, Bollinger Bands, EMA, ATR, MACD)
- 5-model AI intelligence panel (Hermes, Perplexity Sonar, DeepSeek, Llama, GPT)
- Institutional derivatives data from Sharpe.ai (funding rates, insider selling)
- Scientific method parameter optimiser using Sharpe ratio as the objective
- Persistent SQLite history database surviving all restarts
- Live web dashboard with auto-refresh
- Telegram trade notifications

3. TRADING STRATEGY & SIGNAL GENERATION
================================================================================

Signal Engine (analysis.py)
----------------------------
The bot generates a (signal, score) tuple for each pair every loop using
two complementary strategies:

MEAN REVERSION PATH (enable_mean_reversion_signals = true)
  - RSI oversold (< 28) → BUY signal
  - RSI overbought (> 65) → SELL signal
  - Score driven by distance from the threshold
  - Works best in ranging/sideways markets

TREND/BREAKOUT PATH (enable_trend_breakout_signals = true)
  - Price above Bollinger upper band + RSI ≥ 55 → BUY
  - Price below Bollinger lower band + RSI ≤ 45 → SELL
  - Works best in trending markets

The stronger signal wins (highest absolute score takes priority).
ATR (Average True Range) and Williams %R provide small boosts when confirming.

Score Range: −50 (strongly bearish) to +50 (strongly bullish)
Positive = bullish bias, Negative = bearish bias

Indicators Used
---------------
- RSI (14-period)                    Momentum oscillator
- Bollinger Bands (20-period, 2σ)    Volatility bands
- SMA-20 / SMA-50                    Trend direction
- EMA-9 / EMA-21 (optional)          Short-term trend filter (disabled)
- ATR (14-period)                    Volatility for dynamic stops
- Williams %R                         Overbought/oversold confirmation
- MACD (optional, multi-timeframe)   Trend confirmation

Price history: 200 candles per pair, seeded from 15-minute Kraken OHLC data
on startup so indicators work immediately without a warm-up delay.

Trade Execution Flow (per loop)
--------------------------------
1. Get live price via WebSocket (REST fallback if WS stale)
2. Generate signal + score for each pair
3. Select best pair (highest absolute score)
4. Apply intelligence-adjusted buy gate:
   effective_min = min_buy_score + (intelligence_score × score_weight × -1)
5. Check all filters (regime, volume, drawdown, cooldown, position limits)
6. If BUY: check no open short exists on same pair → execute buy
7. If SELL: check open position → check profit target → execute sell
8. If SELL + no position + shorts enabled: open short

Shorting
--------
Shorts are opened when a SELL signal fires but no long position exists.
The bot uses Kraken's margin feature (leverage 2x, paper mode simulated).
A short on XBTEUR means: sell BTC now, buy back cheaper later = profit if
BTC falls.

Short positions are tracked separately from longs in the state files and
are properly restored after restarts.

Multi-Pair Selection
--------------------
Pairs traded: XBTEUR, XETHZEUR, SOLEUR, XXRPZEUR
On each loop, all four pairs are scored and the best signal wins.
auto_select_pair = true means the bot picks whichever pair has the
strongest signal, not a fixed pair.

Stop Loss Mechanisms (3 layers)
---------------------------------
1. SOFT STOP (stop_loss_percent): Regular exit when price falls X% from entry
2. HARD STOP (hard_stop_loss_percent): Force-close even if soft SL not triggered
3. BREAK-EVEN STOP: After price rises 0.5%, move stop to entry price
   (enable_break_even = true, break_even_trigger_percent = 0.5)

The hard stop fires at 4%, soft at 5% — so hard stop always triggers first
in live mode, acting as a backstop.

Position Sizing
---------------
Trade amount = available_balance × (allocation_per_trade_percent / 100)
In paper mode: 10% of €100 = €10 per trade
In live mode: 50% of available EUR balance per trade

Intelligence-Adjusted Buy Gate
-------------------------------
The AI panel score adjusts the minimum buy score threshold:

  effective_min_score = min_buy_score - (intelligence_score × score_weight)

Example: min_buy_score=8, AI score=-3, weight=0.5
  → effective_min = 8 - (-3 × 0.5) = 8 + 1.5 = 9.5

Bullish AI (score=+4) → effective_min = 8 - 2 = 6 (easier to buy)
Bearish AI (score=-4) → effective_min = 8 + 2 = 10 (harder to buy)

5. AI INTELLIGENCE PANEL
================================================================================

Five AI models run in parallel every 10 minutes via OpenRouter and OpenAI.
Each receives a detailed market context prompt and returns a score −5 to +5.
Scores are weighted and averaged into a combined intelligence_score.

MODEL PANEL
-----------
Model               Provider        Weight  Specialisation
------------------  --------------  ------  ---------------------------------
Hermes 3 70B        OpenRouter      1.0×    Strategy & positioning
Perplexity Sonar    OpenRouter      1.5×    LIVE WEB SEARCH — real-time news
DeepSeek R1         OpenRouter      1.0×    Technical pattern reasoning
Llama 3.1 8B        OpenRouter      0.75×   Fast sentiment cross-check
GPT-4o-mini         OpenRouter      1.0×    General market sentiment

Perplexity Sonar gets 1.5× weight because it can search the live internet
for breaking news, regulatory updates and market events in real time.

What Each Model Receives
------------------------
Every model prompt includes:

MARKET DATA:
  - Crypto Fear & Greed Index (last 3 days trend)
  - BTC dominance %
  - Global market cap 24h change
  - Recent crypto news headlines

SHARPE.AI INSTITUTIONAL DATA:
  - Per-coin funding rates (BTC/ETH/SOL/XRP)
  - Funding rate signal scores (contrarian indicator)
  - Market-wide insider selling signal
  - Total perpetual OI ($33B+ tracked)
  - OI-weighted funding rate

BOT PERFORMANCE CONTEXT:
  - Current Sharpe ratio and verdict (success/failure/neutral)
  - Last 10 trades with outcomes (WIN/LOSS/FLAT + P&L)
  - Last 6 AI panel calls and whether they were right
  - Last 4 Sharpe.ai funding rate snapshots (trend)
  - Last 5 optimizer decisions (what changed, Sharpe impact)
  - Open positions (longs and shorts)
  - Current technical signals per pair
  - Account balance

RECENT HISTORY (from SQLite DB):
  - All past trades with outcomes
  - All past AI decisions correlated with market outcomes
  - Funding rate trend over time
  - Optimizer experiment results

Score Interpretation
--------------------
+5  Very bullish — strongly recommend increased exposure
+3  Bullish — market conditions favour longs
+1  Mildly bullish — slight upward bias
 0  Neutral — no clear direction
-1  Mildly bearish — slight downward bias
-3  Bearish — market conditions favour shorts/reduced exposure
-5  Very bearish — strongly recommend reduced/no exposure

Combined score adjusts the buy gate. The bot doesn't blindly follow AI —
the AI score is one input among many, weighted at 0.5× in paper mode.

Intelligence Log
----------------
Every AI panel call is logged to:
  - data/intelligence_log.jsonl (text log)
  - data/bot_history.db (ai_panels table)

After each trade closes, the outcome (WIN/LOSS + P&L) is written back
into the most recent AI panel entry, creating a feedback loop so models
can see whether their past calls predicted correctly.

6. SELF-LEARNING & OPTIMISATION SYSTEM
================================================================================

The bot has two self-improvement mechanisms:

A) SCIENTIFIC METHOD PARAMETER OPTIMISER
-----------------------------------------
File: core/param_optimizer.py

Uses the scientific method: change one variable at a time, measure the
outcome, keep or revert.

Process:
1. Establish a Sharpe ratio baseline from closed trade history
2. Select the next parameter to test from a rotation list
3. Change it by exactly ONE step (up or down)
4. Wait for MIN_EVAL_TRADES (10) new closed trades
5. Calculate new Sharpe ratio
6. If improved → KEEP the change, update baseline
7. If not improved → REVERT the change, try next parameter

Parameters cycled through:
  min_buy_score              Step: 2.0   Range: 5–50
  take_profit_percent        Step: 0.25  Range: 1–6%
  stop_loss_percent          Step: 0.5   Range: 1.5–10%
  mr_rsi_oversold_threshold  Step: 2.0   Range: 20–45
  mr_rsi_overbought_threshold Step: 2.0  Range: 50–75
  volume_filter_min_ratio    Step: 0.1   Range: 0.2–2.0
  allocation_per_trade_percent Step: 5.0 Range: 5–70%

Success criteria: Sharpe ratio ≥ 3.0
Failure trigger:  Sharpe ratio < 1.0

All experiments are recorded in:
  - data/optimizer_state.json (current state, survives restarts)
  - data/bot_history.db (optimizer_decisions table, full history)

The optimizer runs after every 10 closed trades (configurable via
eval_every_n_trades in config.toml [optimizer] section).

B) AI PANEL CALIBRATION (Learning from History)
-------------------------------------------------
The AI models receive their own historical decisions and outcomes in every
prompt. This creates a feedback loop:

"Last time Hermes scored -4 (bearish), BTC dropped 2% — correct call."
"Last time Sonar scored +3 (bullish), ETH rose 1.5% — correct call."
"Funding rate was -1 for SOL, then SOL rose 3% — contrarian signal worked."

Over time, models can calibrate their scoring based on what actually
happened in this specific bot's trading history with these specific pairs.

This is different from traditional ML — the learning happens through the
models' contextual understanding of the provided history, not weight updates.

    >5% APY    → signal -2 (slightly crowded longs — mild bearish)
    Neutral    → signal  0
    <-5% APY   → signal +2 (shorts crowded — mild bullish)

8. PERSISTENT HISTORY DATABASE
================================================================================

File: data/bot_history.db (SQLite, on Railway persistent volume)
Module: core/history_db.py

The database retains up to 5,000 rows per table and prunes older entries
automatically. All data survives Railway redeploys and container restarts.

TABLE: trades
-------------
id, ts, type, pair, qty, price, pnl_eur, balance_after, paper_mode, reason
Populated: On every trade execution (BUY/SELL/SHORT_OPEN/SHORT_CLOSE)
Purpose: Full trade audit trail, feeds into AI context

TABLE: ai_panels
----------------
id, ts, combined_score, hermes_score, sonar_score, deepseek_score,
llama_score, gpt_score, hermes_text, sonar_text, sharpe_at_time, market_outcome
Populated: Every 10-minute AI intelligence refresh
market_outcome: Updated retrospectively when next trade closes (WIN/LOSS)
Purpose: Lets AI models see whether their past calls were right

TABLE: sharpe_snapshots
-----------------------
id, ts, btc_funding_score, eth_funding_score, sol_funding_score,
xrp_funding_score, insider_signal, total_oi_usd, oi_weighted_funding
Populated: Every AI panel refresh (when Sharpe.ai data is fetched)
Purpose: Funding rate trend history, shows how signals evolved

TABLE: bot_snapshots
--------------------
id, ts, loop, balance_eur, trade_count, sharpe_score, sharpe_verdict,
regime, intelligence_score, btc_signal, eth_signal, sol_signal, xrp_signal, paper_mode
Populated: Every 10 loops (~10 minutes)
Purpose: Performance tracking, balance history

TABLE: optimizer_decisions
--------------------------
id, ts, param_key, section, old_value, new_value, sharpe_before, sharpe_after, verdict
Populated: When the optimizer keeps or reverts a parameter change
Purpose: Full experiment audit trail

How History Feeds Back Into AI Prompts
---------------------------------------
On every intelligence refresh, core/history_db.py::build_history_context()
reads the last N entries from each table and formats them as a text block
injected into every model's prompt before they score the market.

This means models see:
- Whether past bearish/bullish calls preceded actual price movements
- Whether funding rate signals proved accurate
- Which parameter changes improved or hurt Sharpe
- What trades have been placed and their outcomes

[ ] Run paper mode for minimum 1-2 weeks, review Sharpe score
[ ] Confirm Sharpe > 1.5 over 30+ trades before going live
[ ] Regenerate OpenRouter API key (sk-or-v1-...) at openrouter.ai/keys
[ ] Regenerate Sharpe.ai API key at sharpe.ai

[ ] Verify Kraken account at Intermediate level
[ ] Confirm API key permissions: Query Funds, Create Orders, Query Orders,
    Query Trades, Query Ledger, Cancel Orders
[ ] Apply for margin trading on Kraken if shorting desired (1-2 business days)
[ ] Review config.toml settings (currently: 50% allocation, 5% SL, 4% hard stop)

To activate live mode:
  Railway → Service → Variables → remove BOT_ARGS variable → Update → Redeploy

The bot will automatically use config.toml (live config) and real Kraken balances.

Revenue model:
  The bot profits from the spread between entry and exit prices, minus fees.
  Kraken maker fee: 0.16%, taker fee: 0.26% (at lowest tier)
  Each round trip (buy + sell) costs ~0.42% in fees.
  Therefore: take_profit_percent must exceed 0.42% just to break even.
  Current setting of 2% TP gives ~1.58% gross profit per winning trade.

13. KNOWN ISSUES & FUTURE IMPROVEMENTS
================================================================================

Known Issues
------------
1. Windows fcntl lock: The single-instance lock and order lock both rely on
   fcntl which is Linux-only. On Windows (development machine), these are
   no-ops. On Railway (Linux), they work correctly.

2. kelly_fraction: Calculated but not used in position sizing. Dead code —
   could be wired into allocation_per_trade_percent calculation.

3. get_crypto_holdings() hardcodes 11 asset mappings. Adding new pairs
   requires updating this dict manually.

4. Hermes 3 70B occasionally ignores the score format instruction and
   returns conversational text. The score parser falls back to extracting
   the first number in the response, which may be inaccurate.

5. Score zeroing: All pair scores sometimes show 0.00 for extended periods.
   This happens when RSI is in the neutral zone (30-70) and price is inside
   the Bollinger Bands simultaneously. The bot holds during these periods.

Planned Improvements
--------------------
- Volatility-based TP advisor: Calculate achievable TP% from recent ATR
  and display suggested parameters on dashboard
- Sharpe.ai futures data: Use long/short ratio data for additional signal
- More pairs: Add ADAEUR, DOTEUR when bot proves profitable on current 4
- Backtesting integration: Run new parameter combinations against historical
  data before the optimizer deploys them live
- WebSocket order execution: Currently uses REST for orders (slow);
  WebSocket order placement would reduce execution latency
- Portfolio rebalancing: Prevent over-concentration in one pair
- Notification improvements: Daily P&L summary, Sharpe score alerts
